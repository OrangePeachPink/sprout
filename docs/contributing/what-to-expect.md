# What to expect when you contribute

A small team keeps Sprout, and we want your time here to be good — so here's plainly how we work with
contributors. No surprises, no hoops.

## Claiming an issue

You don't need permission to start. Comment if you'd like to raise your hand, but an issue is really yours
the moment a **draft PR shows progress**. If you tell us you've read the code and you're taking it, we'll
assign it to you so everyone can see it's yours.

**New here? One at a time.** Pick a single issue, land it, then come back for the next — it keeps things calm
for you and clear for the rest of us.

**If life happens** and it goes quiet for a week or so, we'll gently open it back up for someone else — no
worries at all, and you're always welcome back.

## If two PRs land for the same issue

It happens, and it's genuinely fine. The **first PR that meets the bar wins**; the other we close warmly,
with a redirect to something else to sink your teeth into — never silently, and never as a mark against your
work.

## What review looks like

A human maintainer reads **every** diff. You'll get specific, kind feedback and a clear path forward — not a
rubber stamp, and not a black hole. (Our very first outside PR turned up a real bug in *our* code, and the
back-and-forth made the change better. That's the point — we're building this together.)

**Your first PR's checks wait for a maintainer to approve the run.** That's standard GitHub safety for a
first-time contributor — not a sign anything's wrong with your work.

## Timing

We aim to reply within a day. If we're slow, that's on us, not you — a friendly nudge is always welcome.
