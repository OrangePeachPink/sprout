## 2. Today's event — readiness & how to run it

**The event:** the skylight shaft lands on the bench ~13:00–14:00 local, hitting all four co-located probes
at once. First real **diurnal signature** in the data, seen simultaneously on 4 probes in identical soil.

### 2.1 The capture is already live — the main job is *don't break it*

A live logger (`tools/logger/plants_logger.py`, **PID 45828** as of last night) owns **COM6** and is writing
`logs/plants_esp32_f4e9d4_*.csv`. The single biggest risk to a clean all-day dataset is **resetting the
ESP32**, because (B5, data-team's top continuity note) **every reopen of COM6 resets the board and starts a
new session/file**. So, today:

- **Do not** open the PlatformIO/Arduino serial monitor on COM6, reflash, or restart the logger. Any of those
  severs the day into a new session.
- **Keep the host awake** — disable sleep for the day. A USB suspend stops the capture.
- Daily midnight rotation is host-side only (new file, **same** device session) and is fine — it does not
  reset the board. The B8 LFS archiver gzips closed segments automatically.
- To watch it live without touching the capture, use the **read-only** server in a separate terminal:

  ```text
  python tools/analytics/serve.py        # http://127.0.0.1:8765 — re-parses logs/ on each Refresh
  ```

  It never writes the logs and never touches the serial port.

### 2.2 What to look for (and why it matters downstream)

- **Cross-probe agreement on the transient.** Four probes, same soil, same light step → the cleanest possible
  read on per-probe variance. This is the exact input C1 (per-channel calibration) needs.
- **First diurnal bump** in E3's forecast/diurnal feature — does the light event show as a coherent raw
  excursion across all four, or just thermal noise on the ADC?
- The dashboard intentionally omits day/night shading ("needs a real light schedule"). Today gives us a
  *known-time* light input we can annotate later.

## 3. Firmware roadmap (prioritized)

Ordering principle: **make watering correct and safe before it's possible.** Today's read-only run is the
data that informs P0.

### P0 — Before any pump runs (safety-gating; flagged by the data review)

1. **A2 — reconcile the band boundaries against real soil.** Two concrete risks the data team raised:
   - **Dry-edge fail-to-water (HIGHEST).** `air-dry ≥ 2760` means a genuinely parched pot reading ~2800–2920
     classifies as "air-dry / not in soil" — a *diagnostic* band that **never triggers watering**. Harmless
     today (read-only); a silently un-watered plant once pumps are live. Fix options: lower the air-dry
     threshold, or add a "very dry but still in soil" guard so a real dry pot still waters.
   - **Wet split is below the noise floor.** Saturated soil, pure water, and the wet rail are one cluster
     (±270). Treat everything under ~1150 as a single "too wet / check probe" condition; don't pretend the
     submerged/overwatered split is trustworthy. Lean on `confirm_ms` to debounce.
   - Use **today's 4-probe day** (incl. the skylight transient) as the reconciliation dataset. Decision needed
     from you (see §5).

2. **A1 — wire the health veto into the live supervisor.** Already built + unit-tested; just not in `main.cpp`.
   Fold the veto in, set `max_health_warn ≈ 3`, and surface `irrig_health_warn()` in the serial banner so a
   floating/disconnected probe announces itself instead of quietly reading "dry" and tripping a pump. A1 and
   the A2 dry-edge interact: a floating probe reads "dry"; a real dry pot reads "air-dry" — both must be
   handled before watering is trusted.

### P1 — Robustness & data honesty

1. **B5 — stop the ESP32 reset-on-connect (continuity).** The data team's #1 continuity gap. Either disable
   DTR/RTS auto-reset in the logger's serial open, or move to permanent-install wiring. Until then, the
   operating discipline in §2.1 is the mitigation.
2. **`value=pct` column decision (B2/C2).** The firmware still emits the legacy linear map as `value`/`unit=pct`
   (`main.cpp:167`). It *looks* authoritative but is **not** VWC and the data team flags it as a misuse magnet.
   Producer-side call (mine to implement once you decide): drop it, null it until real calibration exists, emit
   the band index instead, or keep it but rename/document as the legacy index.

### P2 — Toward actual watering (the point of the build)

1. **D1 — pump logging + relay bring-up.** Relays are wired-but-commented (`config.h`, active-LOW CW-022 on
   GPIO 25/26/27/32). Emit `record_type=pump` rows from the FSM (start/stop/duration, dose index, trigger,
   pre/post readings, faults) so the dataset finally has the actuator half. Gated behind P0. Bench-verify one
   relay/pump on one channel before four.

### P3 — Responsiveness & local awareness

1. **A3 — fast health/liveness tick** (1–5 s) decoupled from the 30 s log cadence, feeding D3 (OLED/e-ink/LED
   status) and D4 (WiFi served page). Respect the no-sample-during-pump ADC gate.

### Continuous — Calibration

1. **C1 — per-channel calibration** from the co-located runs. Today's data is the first real input; run the
   same-probe/same-water cal with **all neighbors powered** so it captures cross-coupling, not just an isolated
   probe.
